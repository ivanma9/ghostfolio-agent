# P0 Sidebar UX: Clickable Holdings + Alerts Section

**Date:** 2026-03-01
**Status:** Design approved

## Problem

The sidebar is passive — it shows portfolio data but doesn't drive action. Holdings aren't clickable, alerts from the alert engine have zero UI presence, and capability discoverability is low. Insights generated by the agent (morning briefings, conviction scores, holding deep dives) are ephemeral chat messages that scroll away.

## Solution

Two changes that make the sidebar an active decision-making tool:

1. **Clickable holdings** — tapping a holding in the sidebar sends "Tell me about {SYMBOL}" to the chat
2. **Alerts section** — a persistent "Needs Attention" section that surfaces the alert engine's output visually

## Design

### 1. Clickable Holdings (TopHoldings + AllocationChart)

**Behavior:**
- Each holding row in TopHoldings becomes a `<button>`
- On click, calls `onHoldingClick(symbol)` which calls `onSend("Tell me about {SYMBOL}")`
- AllocationChart donut segments also become clickable with the same behavior
- Props chain: `App.tsx` passes `onSend` → `Sidebar` → `TopHoldings` / `AllocationChart`
- Mobile: holdings remain non-interactive (sidebar hidden on mobile)

**Visual treatment:**
- Default: current look + `cursor-pointer`
- Hover: `bg-slate-50/60` background wash, symbol text shifts gray-900 → indigo-600, 2px left border fades in (indigo-400), small right-chevron icon fades in at right edge (`opacity-0` → `opacity-100`)
- Allocation chart: hovered segment scales out 4px from center via Recharts `activeShape`, cursor pointer
- All transitions: 150ms ease-out
- Allocation progress bar subtly brightens on hover (opacity shift)

### 2. Alerts Section in Sidebar

**Data flow:**
- `ChatResponse` backend model extended with `alerts: list[AlertItem]` field
- Each `AlertItem`: `{ symbol, condition, message, severity }` where severity is `"warning"` or `"critical"`
- Alert engine already produces alerts — we capture them as structured objects alongside the existing text prepend
- LLM-facing behavior unchanged (alerts still prepended to user message text)
- Frontend `useChat` hook stores `activeAlerts: AlertItem[]` in state
- Merge logic: key by `symbol:condition`, newer alerts replace older ones
- Alerts persist for the session (backend 24-hour cooldown prevents duplicate fires)

**Severity mapping:**
- `earnings_proximity` → `"warning"` (amber)
- `big_mover` → `"warning"` (amber)
- `low_conviction` → `"critical"` (red)
- `analyst_downgrade` → `"critical"` (red)

**Placement:** Between PortfolioValue and AllocationChart — highest-visibility real estate after the total value.

**Visual treatment:**
- Section hides entirely when zero active alerts (no empty state)
- Card: subtle warm background `bg-amber-50/40` with `amber-100` border
- Header: "Needs Attention" in uppercase tracking style with a 6px pulsing dot — amber-400 for warnings only, red-400 if any critical alert present
- Each alert: compact row with bold symbol, short condition description, 3px colored left border (amber for warning, red for critical)
- Critical alerts sort above warning alerts
- Max 3 visible, "+N more" expandable for overflow
- Each alert row is clickable — sends "Tell me about {SYMBOL}" same as holdings
- Hover treatment matches holdings: subtle background wash, chevron fade-in
- 12px extra gap between Alerts section and surrounding sections for breathing room

### 3. Backend Changes

- Add `AlertItem` schema: `symbol: str, condition: str, message: str, severity: Literal["warning", "critical"]`
- Add `alerts: list[AlertItem]` field to `ChatResponse`
- In `chat.py`: after alert engine runs, collect fired alerts as structured `AlertItem` objects and include in response alongside existing text prepend behavior
- No new endpoints. No new background processes.

### 4. Frontend State & Wiring

**New type:**
```typescript
interface AlertItem {
  symbol: string
  condition: string
  message: string
  severity: 'warning' | 'critical'
}
```

**useChat hook:**
- New state: `activeAlerts: AlertItem[]`
- On chat response with `alerts`, merge into `activeAlerts` keyed by `symbol:condition`
- Expose `activeAlerts` alongside `messages` and `isLoading`

**App.tsx:**
- Pass `activeAlerts` from `useChat` to `Sidebar`
- Pass `handleSend` to `Sidebar` as `onHoldingClick`

**Sidebar props additions:**
- `alerts: AlertItem[]`
- `onHoldingClick: (symbol: string) => void`
- Passes `onHoldingClick` to `TopHoldings`, `AllocationChart`, and new `AlertsSection`

### 5. Visual Refresh

**Holdings interactivity signals:**
- 2px left border: transparent default → indigo-400 on hover
- Background: transparent → `bg-slate-50/60` on hover
- Symbol color: gray-900 → indigo-600 on hover
- Right chevron: `opacity-0` → `opacity-100` on hover
- All transitions 150ms ease-out

**Alerts visual weight:**
- 3px left border (thicker than holdings hover border) for visual hierarchy
- Warm amber tones distinguish from cool indigo holdings
- Pulsing dot draws eye to active alerts
- Critical (red) alerts sort before warning (amber) alerts

**Allocation chart:**
- Segment hover: scale out 4px from center
- Cursor pointer on segments
- Existing Recharts tooltip preserved
